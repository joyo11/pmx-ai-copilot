# DR-002: M1 kickoff — what ships first, what gets faked

**Date:** 2026-07-09
**Status:** Decided (pre-agent-report)

## Question
The user tried to "create a project" and "test the chatbot" — neither exists yet. M1 (upload → extract → embed → risk → chat) is the whole point of the product. Where do we start, and what do we cut to ship a real demo fast?

## Round-table

- **Vic (CEO):** "Cut everything except the chat + one document upload. The demo story is 'upload a PDF, ask about the project.' That's the one screen a stranger needs to see. Everything else (schedules, budgets, RFI classification) is M2. If M1 doesn't produce a chatbot that answers real questions, we've built nothing."
- **Rachel (Sr EM):** "Agree, but be realistic. M1 as written is 5 days of work. If we split it into 1.5 days for 'upload + chat with citations' and defer the rules-based risk engine to M2, we can ship the demo screen this weekend. The risk engine is the fanciest thing but the least demo-able for the first pass."
- **Kenji (Staff):** "Sequence: (1) project CRUD, (2) doc upload → PyMuPDF text extraction, (3) chunk + embed → pgvector, (4) chat endpoint that streams over SSE with retrieved context. Everything else waits. Don't build the risk engine until we have retrieval that returns the right chunks — otherwise we're building on sand."
- **D Guy:** "Two things a PM will notice immediately if we skip: (a) the answer must cite the PDF page, not just quote it — construction PMs need to open the file to verify. (b) The chunk boundaries need to respect page breaks so citations are correct. If we ship without citations, they'll dismiss it as a demo toy."
- **Priya (Jr):** "The dashboard says 'M0 shipping. Upload → risk engine → chat streams arrive in M1' — that's a promise. When I sign up and land on the dashboard, I need at least ONE 'create a project' button that works. Even if the risk engine is fake, let me feel like the app is alive."
- **Milo (UI):** "The chat screen needs a split layout the moment M1 ships — conversation on the left, sources panel on the right. If sources aren't there on day one, users think it's just ChatGPT."

## Decision

**M1 v1 scope (target: 2 days, not 5):**

1. **Projects: minimum viable CRUD.** Create + list + detail. No members, no advanced filters, no health score computation. Just a "New project" modal that takes {name, client, sector} and writes to Postgres. Existing empty state on `/projects` gets a working button.
2. **Document upload: PDF only** (defer P6, XER, MPP, Excel, DOCX to M2). PyMuPDF extracts text with page numbers preserved. Chunker respects page boundaries. Embeddings via OpenAI `text-embedding-3-large` land in `document_chunks.embedding` (HNSW-indexed already).
3. **Chat: project-scoped RAG with streaming and citations.** New endpoint `POST /v1/projects/{id}/chat` streaming SSE. Retrieval via pgvector cosine, top-8 chunks. LLM = Claude Sonnet 4.6. Response includes citations `[{document_id, chunk_id, page}]` in the SSE `citation` event. UI: split layout, sources panel on the right shows the highlighted chunks.
4. **Dashboard: real data.** Active project count, total documents ingested, most recent chat session. No health score yet.

**M1 v2 (deferred to M2, was originally M1):**
- Rules-based + LLM hybrid risk engine.
- Schedule/budget/RFI structured parsers.
- Health score with reasoning.
- Notifications.

**Chat depends on retrieval, retrieval depends on ingestion, ingestion depends on projects.** So the build order is: projects → upload → embed → retrieve → chat.

**Dissent:** none material. Rachel wanted us to delay projects CRUD and just seed one demo project — overruled because Priya's onboarding argument is stronger.

**Backend:** the FastAPI on Render needs to actually be deployed before chat can work in prod. That's a blocker for M1 v1 v1. Confirm the QA agent finds Render or docs it's missing.

**Escalation to Shafay:** none required. This is scope tightening within the M1 envelope; the roadmap stays 25 days.
